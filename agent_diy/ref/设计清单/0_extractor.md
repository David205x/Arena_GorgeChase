# 0. extractor 设计清单

## 一、分析

### 1. 这一部分在整个任务里的作用
`extractor` 的作用不是直接决定策略优劣，而是作为观测设计与代码实现之间的中间层。它负责把环境原始 `obs` 中分散、嵌套、跨步相关的信息整理成更稳定、可编程、可复用的状态表达，方便后续观测特征构造、奖励计算和调试分析。

### 2. 设计目标
- 将环境原始字典观测结构化为清晰的对象或状态容器。
- 统一维护当前帧信息、跨步缓存信息和基础派生信息。
- 为 `obs` 和 `reward` 提供稳定的数据访问接口，减少后续模块直接依赖底层原始字段。
- 保持职责克制：以服务观测和奖励设计为主，不过度承载训练策略本身。

### 3. 和其他模块的边界
- 与 `obs` 的边界：`extractor` 负责“整理和提供信息”，`obs` 负责“决定哪些信息进入模型输入”。
- 与 `reward` 的边界：`extractor` 可以提供奖励计算需要的中间状态，但不负责定义奖励权重和奖励逻辑目标。
- 与 `algorithm_structure` 的边界：`extractor` 不决定网络结构，只决定有哪些可供消费的状态表达。
- 与 `workflow` 的边界：`extractor` 不负责训练调度和评估流程，只负责提供稳定的数据层支持。

### 4. 结构性
#### 4.1 extractor 的总体职责
- 以环境原始观测为输入，产出可供 `obs`、`reward`、调试逻辑共同使用的结构化状态。
- 管理“当前帧信息”“跨步缓存”“基础派生信息”三类内容。
- 优先提供稳定、清晰、低歧义的数据表达，而不是在 extractor 内直接实现复杂策略。
- 对信息进行适度加工，但不越界承担最终特征选择与奖励权重设计。

#### 4.2 原始观测封装层
- 将环境原始字典封装为 `RawObs` 类或等价容器，统一解包 `frame_state`、`env_info`、`map_info`、`legal_action` 等内容。
- 对英雄、怪物、宝箱、buff 等对象分别封装为轻量状态类，统一字段命名与访问方式。
- 将环境配置类信息（如最大步数、怪物刷新规则、buff 刷新时间、闪现冷却上限）一并纳入结构化状态，供后续阶段判断与派生逻辑使用。
- 原始封装层以“忠实映射环境”为主，不在这一层混入跨步推理。

#### 4.3 跨步完整状态层
- 在 `RawObs` 之上维护一个 `FullObs` 或等价完整状态对象，负责保存跨步缓存。
- 需要能够同时访问当前帧与上一帧的英雄状态、怪物状态、资源状态、得分状态与地图状态。
- 为 `obs` 中的历史行为反馈、为 `reward` 中的变化型 shaping 提供统一数据来源。
- 跨步完整状态层应支持逐帧 `update`，并在 `reset` 时完整清空。

#### 4.4 地图与探索缓存
- 维护全局探索地图 `map_full`，区分“未探索”“已探索可通行”“已探索不可通行”。
- 基于当前位置与 `21*21` 视野持续更新全局地图缓存。
- 维护地图探索率、当前步新探索格数等统计量，供 `obs` 和 `reward` 共享使用。
- 局部地图相关的原始 6 层输入所需底层信息，也应尽量在 extractor 中提前整理好。

#### 4.5 资源全局状态缓存
- 维护 `treasure_full`，记录所有已发现宝箱的全局状态，并持续更新“未发现 / 已发现未收集 / 已收集”。
- 维护 `buff_full`，记录所有已发现 buff 点的位置与刷新状态，必要时估计 cooldown。
- 支持从“当前可见资源”扩展到“历史已知资源”的统一查询。
- 为 `obs` 中的最近已知资源、发现进度、收集进度提供基础数据。

#### 4.6 英雄与动作后果派生信息
- 提供英雄当前位置、buff 状态、闪现可用性、闪现冷却等基础状态。
- 提供跨步动作反馈所需的派生量，如是否发生有效移动、是否带来新探索、是否获取资源、环境得分是否变化。
- 提供普通移动 8 方向可行性、闪现 8 方向落点、闪现有效性、闪现位移长度等基础动作后果接口。
- 这些派生信息属于 extractor 的职责范围，因为它们强依赖地图规则与跨步比较，适合在数据层统一维护。

#### 4.7 怪物压力派生信息
- 提供最近怪物、第二怪物、怪物数量、怪物速度等基础摘要。
- 提供与英雄的相对坐标、切比雪夫距离等低层派生量。
- 提供支持 `obs` / `reward` 的中层压力信息，如最近怪距离变化、双怪平均距离、夹击趋势、活动空间压缩趋势所需的底层数据。
- 怪物未来动作趋势若实现，应作为可选增强项，不纳入第一版 extractor 必须职责。

#### 4.8 空间与局部地图派生信息
- 基于局部地图与全局已知地图，提供八方向通路长度、可活动空间、开阔度等空间统计。
- 支持死角、短走廊、低开阔区域等局部空间判定所需的基础接口。
- 提供局部地图 6 层构造所需的底层通道数据：障碍/可通行、英雄位置、怪物位置、宝箱位置、buff 位置、历史访问。
- 若后续需要风险投影、探索潜力等派生层，应作为增强项在此基础上扩展。

#### 4.9 阶段与节奏信息派生
- 基于当前步数、怪物数量、怪物加速节点等信息，统一派生当前阶段。
- 统一计算距离下一个关键阶段的倒计时。
- 阶段信息应同时服务于 `obs` 的阶段观测和 `reward` 中的阶段权重 $$\alpha$$ 计算。
- 相关逻辑应集中在 extractor，避免 `obs` 与 `reward` 各自重复判断。

#### 4.10 对 `obs` / `reward` 的接口约定
- extractor 对上层提供稳定字段或稳定方法，而不是让 `obs` / `reward` 直接回到底层原始字典取数。
- 对同一概念尽量只维护一份标准定义，例如“最近怪物距离”“最近已知宝箱距离”“地图探索率”。
- `obs` 与 `reward` 共用的变化型字段应优先由 extractor 统一维护，避免双份实现导致语义漂移。
- 对训练期调试与监控有价值、但不直接进入 `obs` 的状态，也可由 extractor 暴露为辅助接口。

## 二、具体属性

### 1. 必须项

#### 1.1 英雄状态容器
| 名称 | 来源 |
|------|------|
| 英雄 id / 绝对坐标 | 原始 `frame_state.heroes` |
| buff 剩余时间 | 原始 `frame_state.heroes` |
| 闪现冷却时间 | 原始 `frame_state.heroes` |
| 英雄是否可闪现 | 由闪现冷却时间派生 |
| 英雄当前速度 | buff 状态 + 环境规则派生 |

#### 1.2 怪物状态容器
| 名称 | 来源 |
|------|------|
| 怪物 id / 绝对坐标 | 原始 `frame_state.monsters` |
| 怪物是否在视野内 | 原始 `frame_state.monsters` |
| 怪物速度 | 原始 `frame_state.monsters` |
| 怪物与英雄相对方向/距离桶 | 原始字段或坐标差派生 |
| 怪物数量 | 当前怪物列表统计 |

#### 1.3 宝箱与 buff 状态容器
| 名称 | 来源 |
|------|------|
| 资源 id / 绝对坐标 | 原始 `frame_state.organs` |
| 资源类型（宝箱 / buff） | 原始 `sub_type` |
| 资源当前状态 | 原始 `status` + 跨步缓存更新 |
| buff cooldown（若可维护） | 环境刷新规则 + 历史事件派生 |
| 当前可见资源列表 | 当前帧资源对象筛选 |

#### 1.4 原始观测封装层
| 名称 | 来源 |
|------|------|
| `step_no` | 原始环境观测 |
| `legal_action` | 原始环境观测 |
| `map_view` / `map_info` | 原始环境观测 |
| `frame_state` 解包结果 | 原始环境观测 |

#### 1.5 跨步完整状态封装层
| 名称 | 来源 |
|------|------|
| 当前帧完整状态 | 当前 `RawObs` |
| 上一帧英雄状态 | 上一帧缓存 |
| 上一帧得分 / 资源 / 怪物数量等统计 | 上一帧缓存 |
| 历史位置轨迹 | 跨步 `pos_history` 缓存 |
| `update` / `reset` 生命周期 | extractor 状态管理 |

#### 1.6 地图探索状态缓存
| 名称 | 来源 |
|------|------|
| 全局探索地图 `map_full` | `21*21` 视野持续写回 |
| 当前地图探索率 | `map_full` 统计派生 |
| 本步新探索格数 | 当前帧与上一帧探索差异 |
| 历史访问记录 | 历史位置轨迹 / 访问计数缓存 |
| 局部地图 6 层底图数据 | 当前视野 + 历史缓存联合派生 |

#### 1.7 宝箱全局状态缓存
| 名称 | 来源 |
|------|------|
| 已发现宝箱全局记录 | 当前可见宝箱持续写入 |
| 宝箱“未发现 / 已发现未收集 / 已收集”状态 | 可见信息 + `env_info.treasure_id` + 跨步更新 |
| 最近已知宝箱 | 宝箱全局状态 + 英雄位置派生 |
| 宝箱发现进度 | 已发现宝箱数量 / 总宝箱数量 |
| 宝箱收集进度 | `treasures_collected / total_treasure` |

#### 1.8 buff 全局状态缓存
| 名称 | 来源 |
|------|------|
| 已发现 buff 点全局记录 | 当前可见 buff 持续写入 |
| buff 是否可获取 / 是否在 cooldown | 当前可见状态 + 历史拾取事件 + 刷新规则派生 |
| 最近已知 buff | buff 全局状态 + 英雄位置派生 |
| buff 发现进度 | 已发现 buff 数量 / 总 buff 数量 |
| buff 获取事件 | `collected_buff` 增量或位置邻接 + buff 状态派生 |

#### 1.9 闪现落点与有效性基础接口
| 名称 | 来源 |

| 8 方向闪现有效性 | 相对位移是否为零派生 |
| 8 方向闪现位移长度 | 相对位移长度派生 |
| 普通移动 8 方向可行性 | 局部地图 + 动作规则派生 |

#### 1.10 英雄动作反馈缓存
| 名称 | 来源 |
|------|------|
| 上次动作是否产生有效移动 | 当前帧与上一帧英雄位置比较 |
| 上次动作是否带来最近怪距离增加 | 当前帧与上一帧最近怪距离比较 |
| 上次动作是否拾取资源 | 宝箱 / buff / 分数增量比较 |
| 上次动作是否带来新探索 | 当前帧与上一帧探索量比较 |
| 上次动作环境收益变化 | 总分 / 步数分 / 宝箱分增量 |

#### 1.11 怪物压力基础接口
| 名称 | 来源 |
|------|------|
| 最近怪物 / 第二怪物 | 当前怪物列表按距离排序 |
| 最近怪物切比雪夫距离 | 英雄与怪物坐标派生 |
| 最近怪距离变化 | 当前帧与上一帧比较 |
| 双怪平均距离 | 当前双怪距离统计 |
| 夹击趋势计算所需底层量 | 双怪方向关系 + 距离派生 |

#### 1.12 空间统计与局部判定接口
| 名称 | 来源 |
|------|------|
| 八方向通路长度 | 局部 / 全局地图通行结构派生 |
| 可活动空间 | 局部邻域或扇区格子统计 |
| 通路开阔度 | 局部连通格统计 |
| 死角 / 走廊 / 低开阔判定基础量 | 局部地图结构派生 |
| 活动空间较上步增减 | 当前帧与上一帧空间统计比较 |

#### 1.13 阶段与节奏接口
| 名称 | 来源 |
|------|------|
| 当前阶段 | 当前步数 + 怪物数量 + 怪物加速节点派生 |
| 距离下一个关键阶段倒计时 | 当前步数与关键节点差 |
| 第二只怪物是否已出现 | 怪物数量派生 |
| 怪物是否进入加速阶段 | 当前步数与加速步数比较 |
| reward 阶段权重 $$\alpha$$ 所需输入 | 当前阶段 / 当前步数派生 |

#### 1.14 对 `obs` / `reward` 的统一访问接口
| 名称 | 来源 |
|------|------|
| `obs` 所需结构化字段访问 | extractor 标准属性 / 标准方法 |
| `reward` 所需变化型字段访问 | extractor 标准属性 / 标准方法 |
| 调试与日志分析辅助字段访问 | extractor 暴露的辅助接口 |
| 统一命名与单一语义来源 | extractor 内部约束 |

### 2. 候选增强项

#### 2.1 怪物动作趋势预测接口
- 基于怪物历史轨迹或寻路规则复现，预测下一步方向。
- 该项对怪物压力观测有价值，但实现正确性要求较高，暂不作为第一版必须项。

#### 2.2 下采样全局地图接口
- 将 `map_full` 或其多通道扩展下采样为 `128*128*n` 的候选输入来源。
- 该项主要服务于后续网络增强，不是当前 obs 第一版所必需。

#### 2.3 风险投影与探索潜力接口
- 在局部地图 6 层基础上额外派生威胁热度、探索潜力、候选探索点等高级通道。
- 适合作为局部地图增强项，而不是一开始就进入基础 extractor。

#### 2.4 更细粒度的访问统计
- 例如局部 `3*3` / `5*5` 访问次数、短窗口重复路径统计等。
- 主要服务于低效移动惩罚与分析，可在 reward 具体实现阶段按需补充。

#### 2.5 训练分析专用辅助字段
- 例如终局死法标签、真实全局危险统计、资源机会分析标签等。
- 这类字段可供训练监控或实验分析使用，但不应和正式 obs / reward 的主逻辑混用。
