# 0. extractor 设计清单

## 一、分析

### 1. 这一部分在整个任务里的作用
`extractor` 的作用不是直接决定策略优劣，而是作为观测设计与代码实现之间的中间层。它负责把环境原始 `obs` 中分散、嵌套、跨步相关的信息整理成更稳定、可编程、可复用的状态表达，方便后续观测特征构造、奖励计算和调试分析。

### 2. 设计目标
- 将环境原始字典观测结构化为清晰的对象或状态容器。
- 统一维护当前帧信息、跨步缓存信息和基础派生信息。
- 为 `obs` 和 `reward` 提供稳定的数据访问接口，减少后续模块直接依赖底层原始字段。
- 保持职责克制：以服务观测和奖励设计为主，不过度承载训练策略本身。

### 3. 和其他模块的边界
- 与 `obs` 的边界：`extractor` 负责“整理和提供信息”，`obs` 负责“决定哪些信息进入模型输入”。
- 与 `reward` 的边界：`extractor` 可以提供奖励计算需要的中间状态，但不负责定义奖励权重和奖励逻辑目标。
- 与 `algorithm_structure` 的边界：`extractor` 不决定网络结构，只决定有哪些可供消费的状态表达。
- 与 `workflow` 的边界：`extractor` 不负责训练调度和评估流程，只负责提供稳定的数据层支持。
