# 3. algorithm_structure 设计清单

## 一、分析

### 1. 这一部分在整个任务里的作用
`algorithm_structure` 负责把已经定义好的观测和奖励，映射为一个适合训练和推理的算法与网络实现框架。它的重点不是发明复杂算法，而是确保输入组织、编码方式和策略/价值学习过程与任务结构相匹配。

### 2. 设计目标
- 采用稳定可靠的 PPO 作为训练主算法。
- 让网络输入结构尽量贴近任务本身，而不是把所有信息完全无结构拍扁。
- 区分不同类型输入的编码方式，例如标量、实体、局部地图。
- 为后续扩展留出空间，例如增加更强的实体融合或阶段感知模块。

### 3. 和其他模块的边界
- 与 `obs` 的边界：`obs` 决定输入内容，`algorithm_structure` 决定如何编码和融合这些输入。
- 与 `reward` 的边界：`reward` 决定训练目标，`algorithm_structure` 决定模型如何去拟合这个目标。
- 与 `extractor` 的边界：`extractor` 提供数据层支持，`algorithm_structure` 不直接关心底层原始字段。
- 与 `workflow` 的边界：`algorithm_structure` 不负责训练调度与评估流程，但其好坏要通过 workflow 中的指标与实验结果体现。

### 4. 结构性
#### 4.1 算法选择原则
- 复用 `reference_ppo` 的 PPO 主体
- 算法层重点解决“如何稳定训练”，结构层重点解决“如何组织观测并让网络有效利用”。
- 若后续效果不佳，优先检查观测表达、奖励结构与输入组织

#### 4.2 PPO 主体配置
- 延续 `reference_ppo` 中的 PPO 基本实现：GAE、clip objective、advantage normalization、entropy regularization。
- 保留 legal action masking 的接入方式，不单独设计额外动作约束模块。
- 保留 actor 与 critic 共享主干、分头输出的总体结构。
- 价值网络优先沿用参考实现中的分布式 value 预测方式，第一版不急于改成普通标量回归。
- dual-clip、梯度裁剪、熵系数等稳定训练手段优先保留。

#### 4.3 输入组织总思路
- 采用“结构化向量分支 + 局部地图分支”为第一版主结构。
- 全局信息第一版优先以统计摘要或少量关键坐标进入向量分支，而不是直接上完整全局多通道地图。
- 实体级编码（如怪物、宝箱、buff 等 token 化）保留为候选增强方向，而不是第一版强依赖项。
- 社区分享中提到的“元素 token 化 + 类注意力机制”适合处理多对象关系、动态目标切换与资源/风险竞争问题。
- 该方向与本任务中的怪物夹击关系、资源选择关系、探索目标切换具有一定匹配性，但当前阶段更适合作为增强项而不是主结构。
- 第一版优先验证“结构化向量 + 局部地图”是否已足够表达主要决策信息；若后续确认多对象关系仍是瓶颈，再考虑引入关键对象 token 分支及轻量 attention 机制。

#### 4.4 结构化向量分支
- “结构化”不是指对象 token 化，而是指先按语义把高层特征分块组织，再统一进入向量编码器，而不是把所有数字无差别视为同一类输入。
- 用于编码已经具有明确语义、适合直接给网络的高层特征。
- 内部可进一步划分为若干子组，例如英雄自身状态、怪物压力摘要、资源机会摘要、阶段节奏信息、空间统计与动作后果摘要。
- 这些子组在实现上可以是简单拼接后统一编码，也可以先各自做小型 MLP 投影后再合并；第一版优先选择更简单稳定的实现。
- 包含英雄自身状态、历史动作反馈、下一步动作后果摘要。
- 包含怪物压力摘要，如最近怪距离、双怪关系、夹击趋势、空间压缩趋势。
- 包含资源机会摘要，如最近已知宝箱/ buff 方向与距离、探索程度、收集进度。
- 包含阶段与节奏信息，如当前步数、当前阶段、距离关键阶段倒计时。
- 包含局部空间统计，如通路长度、可活动空间、开阔度等派生量。

#### 4.5 局部地图分支
- 以 `21*21*n` 局部地图作为主要空间输入。
- 通过卷积视觉编码器提取局部几何关系，而不是依赖向量特征手工概括所有空间语义。
- 局部地图各层尽量遵循“一层只描述一件事”的原则，避免在同一通道中混合多种来源不同的语义。
- 第一版局部地图层数暂定为 6 层，以控制实现复杂度并保证核心空间语义清晰。
- 这 6 层优先包括：障碍/可通行层、英雄位置层、怪物位置层、宝箱位置层、buff 位置层、历史访问层。
- 第一版优先保证“局部危险、局部空间、局部资源”三类核心通道表达清晰，不要求一次把所有候选通道都接入。
- 怪物威胁投影、探索潜力、候选落点等派生层暂列为后续增强项，待基础版本验证后再决定是否加入。

#### 4.6 全局信息与候选增强结构
- `obs` 中提到的全局地图矩阵 `128*128*n` 暂不作为第一版主输入。
- 第一版优先采用全局探索率、资源发现进度、收集进度、候选探索坐标等摘要信息。
- 若后续发现模型在全局探索调度或远距离资源规划上明显不足，再考虑引入下采样全局图或记忆图分支。
- 实体级编码可作为另一条增强路线：将最近怪物、第二怪物、最近宝箱、buff、候选探索点组织为少量 token 单独编码。

#### 4.7 多路特征融合
- 参考 `reference_ppo` 的“分编码器 + 融合 torso”思路，对不同分支先独立编码再融合。
- 第一版优先采用简单稳定的拼接融合，而不是一开始引入复杂 attention 或 gating 机制。
- 融合后的共享主干负责汇总“局部空间 + 高层语义 + 阶段信息”的联合表示。
- 若后续发现不同模态之间信息竞争明显，再考虑更强的融合模块。

#### 4.8 actor / critic 输出结构
- actor 与 critic 共享编码器与融合主干，仅在输出头分开。
- actor 头输出动作 logits，并在 logits 层接入 legal action mask。
- critic 头输出状态价值，第一版优先沿用参考实现中的值分布预测方案。
- 两个头保持相近复杂度，避免 critic 过弱导致优势估计不稳定。

#### 4.9 第一版实现优先级
- PPO 主体尽量复用 `reference_ppo`，减少无关改动。
- 优先实现“结构化向量 + 局部地图”双分支主结构。
- 暂不把完整全局图作为必要输入。
- 暂不强制引入实体 token 编码、attention 融合、时序模块等额外复杂结构。
- 先验证新的 `obs` 与 `reward` 是否已经带来有效收益，再决定是否继续增加网络复杂度。
