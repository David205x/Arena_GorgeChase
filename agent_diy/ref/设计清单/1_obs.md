# 1. observation 设计清单

## 一、分析

### 1. 这一部分在整个任务里的作用
`obs` 是整个策略学习中最直接影响决策质量的部分。它的任务不是简单复述环境里“有什么”，而是尽量把“当前危险、资源机会、行动空间、阶段压力”表达清楚，让模型更容易学习在拿资源与保命之间切换。

### 2. 设计目标
- 围绕决策需求重构观测，而不是机械堆叠原始字段。
- 同时覆盖四类核心信息：危险、机会、空间、阶段。
- 尽量让模型能直接感知当前局面中的关键矛盾，例如怪物压力、宝箱收益、闪现操作空间和地形安全性。
- 为后续网络结构提供有组织的输入分块，而不是无差别拍扁。

### 3. 和其他模块的边界
- 与 `extractor` 的边界：`extractor` 提供结构化状态，`obs` 决定最终采用哪些状态作为模型输入。
- 与 `reward` 的边界：`obs` 负责表达决策所需信息，但不直接承担行为导向；行为导向主要由奖励承担。
- 与 `algorithm_structure` 的边界：`obs` 不决定网络实现细节，但会影响网络输入分组方式。
- 与 `workflow` 的边界：`obs` 不负责训练流程，但其设计效果要通过 workflow 中的指标进行验证。

### 4. 结构性
#### 4.1 观测设计原则
#### 4.2 英雄自身状态观测

- 当前状态
  - 绝对坐标
  - buff持续时间
  - 闪现是否可用
  - 闪现冷却时间
- 历史行为反馈
  - 有效性：上次动作是否产生有效移动
  - 安全性：与最近怪物的距离是否增加
  - 资源获得：是否拾取到了资源
  - 地图探索：是否带来地图新探索
  - 环境反馈：上次动作带来的环境收益

- 行为后果：对于下一步
  - 普通移动8个方向可行性
  - 各动作是否产生有效位移
  - 闪现的预期8个位置坐标
  - 闪现带来的等价位移长度
  - 各动作的先验偏好

#### 4.3 怪物压力观测

- 怪物单体
  - 是否存在 (阶段)
  - 相对坐标
  - 切比雪夫距离（若能实现行为跟踪可以换成等效步数）
  - 是否为最近怪物
  - 速度
  - 动作趋势下一步方向 (在复现并验证怪物寻路算法准确性后再使用)

- 怪物组合
  - 夹击趋势 (向量夹角 * 距离衰减)
  - 活动空间压缩趋势：由是否在走廊是否开阔、存在多少安全行动方向 联合计算 是否下降


#### 4.4 宝箱与 buff 机会观测

- 引导：根据全局可见资源进行计算

  - 最近宝箱方向和距离
  - 最近buff方向和距离
  - 资源安全性（是否与怪物在同一方向上）
  - 值得探索的坐标

- 统计

  - 当前地图探索程度


  - 当前宝箱、buff发现进度

  - 当前宝箱收集进度


#### 4.5 局部地图与空间观测

- 局部地图矩阵 `21*21*n`
- 全局地图矩阵 `128*128*n` 应该是要下采样的
- 风险相关：根据已知全局地图进行计算(或许21*21视野内？)
  - 通路长度 (八方向动作连续可执行距离)
  - 可活动空间 （划分八个部分可行动的格子数量）
  - 通路开阔度 (八个部分连通的可行动格子数量)


#### 4.6 阶段信息与节奏信息观测

- 当前步数
- 当前阶段(根据怪物行为划分)
- 距离下个重要阶段的倒计时

#### 4.7 观测组织方式

- 在想直接多维度全局地图，将访问次数、动作落点、历史位置、探索情况、各元素位置甚至移动趋势标注，这样或许会更有结构

> 批注：这个想法很有潜力，而且明显已经是在想“如何组织输入”了。不过我建议我们在当前阶段先把它视为一个候选方向，而不是立即定案。因为它已经从“要有什么信息”跨到了“怎么编码这些信息”。
>
> 批注：它的优势是结构统一、空间信息强；风险是：
> - 实现量大
> - 训练成本高
> - 很多高层语义可能被淹没在大图里
> 所以后面也许要讨论一个折中方案：局部图 + 少量全局统计 + 少量关键先验，而不是一开始就把所有东西堆到多通道全局图里。


## 二、具体属性

#### 1.1 英雄自身状态
| 名称 | 来源 |
|------|------|
| 英雄绝对坐标 | `frame_state.heroes.pos` |
| 英雄 buff 持续时间 | `frame_state.heroes.buff_remaining_time` |
| 英雄是否可闪现 | `frame_state.heroes.flash_cooldown` |
| 英雄闪现冷却时间 | `frame_state.heroes.flash_cooldown` |
| 上次动作是否产生有效移动 | 上一帧与当前帧英雄位置差 / 动作结果缓存 |
| 上次动作后最近怪物距离是否增加 | 上一帧与当前帧最近怪物距离比较 |
| 上次动作是否拾取资源 | `env_info.treasures_collected` / `env_info.collected_buff` 增量 |
| 上次动作是否带来地图新探索 | 当前帧与上一帧探索区域比较 |
| 上次动作带来的环境收益 | `reward.reward` 或环境得分增量 |
| 普通移动 8 个方向可行性 | `map_info` + 移动规则派生 |
| 各动作是否产生有效位移 | `map_info` + 动作结果派生 |
| 闪现的预期 8 个位置坐标 | `map_info` + 闪现规则派生 |
| 闪现带来的等价位移长度 | 闪现预期落点与当前位置差 |
| 各动作的先验偏好 | 局部地图 / 怪物压力 / 资源机会综合派生 |

#### 1.2 怪物压力观测
| 名称 | 来源 |
|------|------|
| 怪物是否存在 | `frame_state.monsters` |
| 怪物相对坐标 | `frame_state.monsters[].pos` 与英雄位置差 |
| 怪物切比雪夫距离 | 怪物位置与英雄位置派生 |
| 怪物是否为最近怪物 | 多只怪物距离比较派生 |
| 怪物速度 | `frame_state.monsters[].speed` |
| 怪物动作趋势下一步方向 | 怪物历史轨迹 / 寻路规则复现派生 |
| 怪物夹击趋势 | 两只怪物相对英雄方向关系 + 距离衰减派生 |
| 活动空间压缩趋势 | 局部空间结构 + 怪物位置联合派生 |

#### 1.3 宝箱与 buff 机会观测
| 名称 | 来源 |
|------|------|
| 最近宝箱方向和距离 | 当前可见或历史已知宝箱位置与英雄位置派生 |
| 最近 buff 方向和距离 | 当前可见或历史已知 buff 位置与英雄位置派生 |
| 资源安全性 | 资源方向与怪物方向/距离联合派生 |
| 值得探索的坐标 | 全局探索状态 / 未探索区域分布派生 |
| 当前地图探索程度 | 全图探索统计派生 |
| 当前宝箱、buff 发现进度 | 已知资源数量与总资源数量比较 |
| 当前宝箱收集进度 | `env_info.treasures_collected` / `env_info.total_treasure` |

#### 1.4 局部地图与空间观测
| 名称 | 来源 |
|------|------|
| 局部地图矩阵 `21*21*n` | `map_info` 及其多通道扩展 |
| 全局地图矩阵 `128*128*n` 下采样表达 | 全图探索状态 / 全局地图缓存 |
| 八方向通路长度 | 局部或全局地图可通行结构派生 |
| 可活动空间 | 局部邻域或扇区可行动格数量派生 |
| 通路开阔度 | 连通可行动格数量派生 |

#### 1.5 阶段信息与节奏信息观测
| 名称 | 来源 |
|------|------|
| 当前步数 | `step_no` |
| 当前阶段 | 当前步数 + 第二只怪是否出现 + 怪物是否加速派生 |
| 距离下个重要阶段的倒计时 | 当前步数与关键阶段步数差 |
